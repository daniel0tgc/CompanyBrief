## 4. Auth Rules (NextAuth v5 + Fastify JWT)

- Auth is NextAuth v5 (Auth.js) with Google provider only вЂ” no email/password, no other providers
- `NEXTAUTH_SECRET` / `AUTH_SECRET` is shared between Next.js and Fastify вЂ” never hardcoded
- After verification, Fastify extracts `userId` from the token and attaches it to `req.user`
- Client components never have direct access to the session token вЂ” they use server actions or server component props
- `middleware.ts` in Next.js protects all routes under `/(app)/` вЂ” unauthenticated в†’ `/sign-in`
- `GET /auth/me` on Fastify returns the current user object вЂ” Next.js server components call this to hydrate user state

**NextAuth v5 specifics (differs from v4 вЂ” do not revert these):**
- Session token is **JWE-encrypted** (not signed) вЂ” stored in `next-auth.session-token` cookie
- Fastify MUST verify using `jose` `jwtDecrypt` with key = `SHA-256(NEXTAUTH_SECRET)` вЂ” NOT `jsonwebtoken` (which only handles signed JWTs, not JWE)
- `auth()` returns a **Session object**, NOT the raw token. To pass the token to Fastify:
  ```ts
  import { cookies } from 'next/headers'
  const token = cookies().get('next-auth.session-token')?.value
              ?? cookies().get('__Secure-next-auth.session-token')?.value
  ```
- `trustHost: true` MUST be set in the NextAuth config for any non-localhost deployment (Railway, Vercel, etc.). Without it, all auth routes throw `UntrustedHost` errors.
- Also set `AUTH_TRUST_HOST=1` as a Railway env var on the web service as belt-and-suspenders.
- Register Google OAuth redirect URIs in Google Cloud Console BEFORE testing the OAuth flow:
  - `http://localhost:3000/api/auth/callback/google`
  - `https://[railway-web-url]/api/auth/callback/google`

## 5. Stack-Specific Rules

### Next.js (Frontend вЂ” apps/web)
- App Router only вЂ” no Pages Router patterns
- Server components fetch from Fastify directly (no client-side fetch in server components)
- Client components use `react-query` (`@tanstack/react-query`) for data fetching вЂ” never raw `fetch` in `useEffect`
- All Zustand stores in `apps/web/store/` вЂ” client-side only, never accessed in server components
- Styling: Tailwind CSS + shadcn/ui components вЂ” no inline styles, no CSS modules
- Never use `any` вЂ” all Fastify response shapes have corresponding types in `apps/web/lib/types.ts`
- File-based routing via App Router вЂ” `app/(app)/` for authenticated routes, `app/(auth)/` for unauthenticated

**Next.js App Router deployment rules:**
- Every deployment on Railway MUST expose `GET /api/healthz` returning `{ ok: true }` (no auth).
  Set Railway web service healthcheck path to `/api/healthz` вЂ” NEVER `/` (middleware redirects
  unauthenticated `/` to `/sign-in` with 302, which Railway treats as unhealthy в†’ FAILED deploy).
- The middleware `matcher` must exclude `/api/*` so healthcheck and NextAuth routes are never auth-gated.
- `process.env` variables used inside `middleware.ts` (Edge Runtime) must be set at BUILD TIME or
  added to `next.config.mjs` `env{}` вЂ” runtime-only Railway env vars may not reach Edge middleware.

### Fastify (Backend вЂ” packages/api)
- All routes in `packages/api/src/routes/` вЂ” one file per domain
- Auth verified via the Fastify JWT plugin (verifyNextAuthToken hook) вЂ” never inline verification
- All route handlers typed with Fastify's generic `RouteHandler` вЂ” never untyped `req`/`reply`
- Background jobs via BullMQ вЂ” never setTimeout for async work
- CORS configured to allow only `NEXT_PUBLIC_APP_URL` origin вЂ” wildcard never allowed in production
- Rate limiting via `@fastify/rate-limit` backed by Redis on all public-facing routes

### Database (Drizzle + PostgreSQL)
- All DB access through `packages/api/src/db/repository/` вЂ” never raw queries in routes
- Schema changes require a new migration file вЂ” never edit the DB directly or modify existing migrations
- Fixtures in `packages/api/src/db/fixtures/seed.ts` вЂ” used for local dev only

### AI Agent (Anthropic + BullMQ)
- All Anthropic API calls run inside BullMQ workers (`packages/api/src/jobs/`) вЂ” zero exceptions
- The analysis agent runs section-by-section вЂ” one `messages.create` call per section
- After each section completes, publish a Redis pub/sub event: `analysis:[companyId]` channel
- Tool call results are processed in a loop until `stop_reason === 'end_turn'`
- Agent system prompt is loaded from `packages/api/src/lib/agent/ANALYSIS_CONTEXT.md` at startup вЂ” never hardcoded inline
- Use `claude-sonnet-4-6` for all analysis calls and chat calls

### SSE Streaming
- SSE endpoint: `GET /companies/:id/stream` вЂ” Fastify sets `Content-Type: text/event-stream`
- SSE route subscribes to Redis pub/sub channel `analysis:[companyId]`
- Frontend uses native `EventSource` вЂ” wrapped in a custom hook `useAnalysisStream`
- EventSource is cleaned up in the hook's cleanup function (component unmount)
- SSE events are JSON-encoded: `data: {"type":"thinking","content":"..."}\n\n`
